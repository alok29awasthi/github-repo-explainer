# AI Repository Explainer

AI Repository Explainer is a tool that analyzes any public GitHub repository and generates a **structured explanation of the project** using AI.

Instead of relying only on the README (which is often missing, outdated, or autogenerated), the tool analyzes the **repository structure and file paths** to infer the architecture, technologies, and purpose of the project.

It produces a **structured JSON response + a detailed Markdown explanation** that can be rendered in a frontend UI.

---

# Features

* Explains any public GitHub repository
* Works even if the README is missing or poor
* Analyzes repository structure to infer architecture
* Detects likely tech stack
* Returns structured output for frontend rendering
* Produces a readable Markdown explanation

---

# How It Works

The system analyzes repositories using two sources of information:

### 1. Repository File Structure

The GitHub API is used to fetch the repository tree.

```
GET /repos/{owner}/{repo}/git/trees/{branch}?recursive=1
```

Only **relevant file paths** are kept.

Filtering removes:

* test files
* abstract classes
* non-code files
* documentation files

Example paths used by the AI:

```
src/.../controller/.../UserController.java
src/.../service/.../UserService.java
src/.../repository/.../UserRepository.java
pom.xml
```

These paths allow the AI to infer:

* architecture layers
* technologies
* project structure

---

### 2. README Analysis

If the repository has a README, it is included in the prompt.

However the system accounts for cases where the README is:

* missing
* autogenerated
* outdated
* very minimal

In those cases the AI relies primarily on the **repository structure**.

---

# AI Prompt Strategy

The prompt includes:

* repository file paths
* README contents
* instructions to infer architecture and technologies

The AI returns **strict JSON**.

Example output:

```json
{
  "summary": {
    "name": "OpenRAG",
    "overview": "OpenRAG is a Retrieval-Augmented Generation platform...",
    "problemSolved": "It enables intelligent document search and AI conversations...",
    "techStack": ["Python", "FastAPI", "Next.js", "OpenSearch"]
  },
  "markdownExplanation": "## Overview\nOpenRAG is a platform..."
}
```

The markdown explanation is rendered in the frontend.

# API Endpoint

### Explain Repository

```
GET /repo/explain
```

Parameters

```
owner = GitHub repository owner
repo  = repository name
```

Example

```
/repo/explain?owner=spring-projects&repo=spring-boot
```

Response

```
{
  summary: {...},
  markdownExplanation: "..."
}
```

# Repository Structure Optimization

Large repositories can contain thousands of files.

To keep prompts efficient, the system:

* keeps only **blob file paths**
* removes **test files**
* removes **abstract classes**
* keeps only **core code directories**
* compresses deep paths
* limits the number of paths sent to the AI

Typical reduction:

```
11300 files → ~200 useful paths
```

---

# Why This Approach Works

Large language models can infer architecture from patterns like:

```
Controller
Service
Repository
Config
Application
```

Even without reading the code.

Providing **file paths + filenames** gives strong signals about the project structure.
